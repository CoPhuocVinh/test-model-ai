Implemented the hard-code cleanup and runtime-safety pass for the ai-agent-iot LLM flow.

The main changes are in config.ts, intentParser.chain.ts, deviceAgent.graph.ts, and logger.ts. INTENT_CONFIDENCE_THRESHOLD now actually controls clarification behavior at runtime, repair parsing no longer hard-codes numPredict: 80 or its confidence threshold, and the Ollama/parser knobs are configurable via env (OLLAMA_MODEL, OLLAMA_FORMAT, OLLAMA_TEMPERATURE, LLM_NUM_PREDICT, LLM_REPAIR_NUM_PREDICT, SEARCH_REPAIR_CONFIDENCE_THRESHOLD). Config parsing now fails fast for invalid numeric env values instead of allowing NaN to leak into runtime behavior.

On the runtime side, docker-compose.agent-phase1.yml now lets the parser/runtime knobs be overridden from shell env, .env.example and README.md document the knobs and prompt locations, and both Dockerfiles use npm ci for deterministic installs. I also fixed multi-turn history so pronoun follow-ups use previous history instead of the current user message, moved more heuristic vocabulary into intentParser.constants.ts, and replaced graph console.log calls with a small logger that redacts user messages by default.

Verification: npm run build passes for both ai-agent-iot/ai-gateway and ai-agent-iot/mcp-device-server. npm test also passes on Node v24.14.0: ai-gateway has 4 passing test files / 19 tests, and mcp-device-server has 2 passing test files / 7 tests. docker compose -f docker-compose.agent-phase1.yml config also succeeds.
